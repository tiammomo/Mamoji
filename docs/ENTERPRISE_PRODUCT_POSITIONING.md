# Mamoji 企业版产品定位

> 本文定义 Mamoji 后续产品方向。自本版本起，Mamoji 不再定位为个人或家庭记账软件，而是面向初创公司和小团队的企业经营记账与人员经营管理工具。

## 1. 一句话定位

Mamoji 企业版是面向初创公司团队的轻量经营管理系统，帮助公司统一管理员工生命周期、企业收入、成本支出、税费、预算、现金流和经营风险。

## 2. 目标用户

- 创始人和 CEO：关注现金流、收入、成本、利润、税费压力、公司剩余 runway 和团队规模变化。
- 财务负责人：维护收入、支出、税费、账户、预算、报销、付款和经营报表。
- HR 或行政负责人：维护人员入职、离职、岗位、部门、合同、薪资口径和员工档案。
- 部门负责人：查看本部门预算、成本、项目支出和人员成本。
- 普通员工：提交报销、查看个人资料、处理待办审批。

## 3. 核心价值

- 把人和钱放在同一个经营视图里：人员变化会影响薪资、社保、公积金、福利、设备和办公成本。
- 让初创团队快速看清公司状态：收入、成本、税费、利润、现金余额和 runway。
- 用轻量流程替代复杂 ERP：入职、离职、报销、付款、预算、税费提醒都能用较少配置跑起来。
- 保留审计和权限边界：关键数据有操作记录，敏感财务和人员信息按角色访问。

## 4. MVP 功能范围

### 公司与组织

- 多公司主体：支持同一团队管理多个公司、分公司或异地主体，按公司隔离经营数据。
- 公司基础信息：公司名称、统一社会信用代码、注册地址、行业、币种、纳税人类型。
- 地区与政策画像：维护注册地、经营地、主管税务机关、财年起始月份和政策画像版本。
- 组织结构：部门、岗位、汇报关系、成本中心。
- 公司账户：基本户、一般户、现金备用金、企业信用卡、第三方收款账户。
- 会计期间：按月、季度、年度查看经营数据。

### 人员管理

- 员工档案：姓名、手机号、邮箱、部门、岗位、入职日期、员工状态、合同信息。
- 入职流程：待入职、已入职、试用期、转正。
- 离职流程：离职申请、最后工作日、交接事项、账号回收、设备归还。
- 人员成本：工资、奖金、社保、公积金、个税、福利、设备和招聘成本。
- 权限联动：员工状态变化影响系统登录、审批权限和数据可见范围。
- 员工地点：记录工作地、社保缴纳地和公积金缴纳地，用于匹配地区政策。

### 收入管理

- 收入类型：主营业务收入、项目收入、订阅收入、服务收入、投资收益、其他收入。
- 客户和合同：客户名称、合同金额、合同周期、回款计划、实际回款。
- 应收和回款：记录开票、收款、未收款和逾期。
- 收入分析：按客户、产品、项目、月份统计收入贡献。

### 成本支出

- 成本类型：人力成本、办公租赁、云服务、市场投放、差旅交通、采购、行政、法务财税、研发工具。
- 成本中心：按部门、项目、客户或业务线归集支出。
- 供应商管理：供应商名称、付款账户、合同、发票和付款记录。
- 报销与付款：员工报销、采购付款、周期性支出、付款状态。

### 税费管理

- 税种配置：增值税、企业所得税、个税、附加税、印花税等可配置税种。
- 税费估算：按收入、成本和期间估算应纳税额。
- 申报提醒：按月、季度维护申报状态、截止日期和缴纳状态。
- 税费报表：显示已缴税费、预计税费、未缴税费和税费占收入比例。
- 地区政策：按公司所在地和政策版本维护申报日历、税务提醒和估算依据。

### 预算与经营控制

- 公司总预算：月度、季度、年度成本上限。
- 部门预算：按部门或成本中心设置预算。
- 项目预算：按项目、客户或业务线跟踪预算。
- 预警规则：预算使用率、单笔大额支出、异常成本增长、现金流不足。
- 经营指标：收入、成本、毛利、净利润、现金余额、burn rate、runway。

### 审批与审计

- 审批场景：入职、离职、报销、付款、预算调整、供应商新增。
- 审批角色：提交人、部门负责人、财务、HR、管理员、创始人。
- 审计日志：关键对象的创建、修改、删除、审批、付款和权限变更都留痕。

## 5. 领域模型调整

| 旧个人记账概念 | 企业版目标概念 | 调整说明 |
| --- | --- | --- |
| 用户 | 员工 + 系统账号 | 员工是业务对象，账号是登录凭证，二者需要解耦 |
| 家庭/账本 | 公司 + 经营账本 | 数据边界从个人/家庭转为公司主体 |
| 分类 | 科目/成本类型/收入类型/税种 | 分类需要支持财务口径和经营分析口径 |
| 账户 | 公司资金账户 | 账户归属公司，支持银行账户、现金、企业信用卡和第三方支付 |
| 交易 | 经营流水/凭证/收付款记录 | 交易需要关联客户、供应商、员工、部门、项目和税费 |
| 预算 | 公司/部门/项目预算 | 预算需要按组织和经营维度归集 |
| 定期记账 | 周期性经营事项 | 房租、工资、订阅工具、税费申报等周期事项 |
| 用户管理 | 人员信息与权限管理 | 覆盖入职、离职、档案、角色和访问权限 |

## 6. 建议数据模型

第一阶段建议新增或重构这些核心表：

- `companies`：公司主体。
- `company_groups`：多公司经营主体或集团。
- `company_memberships`：用户在不同公司下的角色与数据范围。
- `regions`：国家、省、市、区县层级地区。
- `policy_profiles`：地区政策画像。
- `policy_versions`：政策版本与来源。
- `policy_rules`：税务、人事、社保、公积金和申报日历规则。
- `company_policy_overrides`：公司级政策覆盖项。
- `departments`：部门和成本中心。
- `employees`：员工档案。
- `employee_locations`：员工工作地、社保地、公积金地及生效期。
- `employment_events`：入职、转正、调岗、离职等人员事件。
- `company_accounts`：公司资金账户。
- `counterparties`：客户和供应商。
- `business_categories`：收入、成本、税费、资产、负债等经营分类。
- `business_transactions`：企业经营流水。
- `budgets`：公司、部门、项目预算。
- `tax_items`：税种、税期、应缴、已缴和申报状态。
- `approval_requests`：审批单。
- `audit_logs`：审计日志。

第二阶段再补充：

- `contracts`：客户合同和供应商合同。
- `invoices`：开票、收票和发票状态。
- `payroll_items`：薪资、奖金、社保、公积金和个税明细。
- `projects`：项目或业务线。

## 7. 权限设计

建议从简单角色开始：

- `founder`：公司所有者，可查看和管理全部数据。
- `finance_admin`：财务管理员，管理账户、交易、税费、预算、报表。
- `hr_admin`：HR 管理员，管理员工档案、入职、离职、人员成本。
- `department_manager`：部门负责人，查看本部门员工、预算和支出。
- `employee`：普通员工，维护个人资料，提交报销和查看自己的审批。
- `viewer`：只读观察者，用于投资人、顾问或外部审计演示。

权限不建议只按整数位掩码长期扩展。企业版应逐步迁移到 `role + permission + scope + company_set` 模型，其中 `scope` 可以是多公司集团、单公司、指定公司集、部门、项目或本人。

## 8. 报表与指标

企业版首页建议优先展示：

- 本月收入
- 本月成本
- 本月利润
- 现金余额
- burn rate
- runway
- 应收金额
- 应付金额
- 预计税费
- 员工人数
- 本月入职/离职人数
- 预算风险

核心报表：

- 经营概览
- 多公司合并经营看板
- 收入趋势
- 成本结构
- 部门成本
- 项目盈亏
- 税费看板
- 人员成本报表
- 现金流预测
- 预算执行报表

## 9. 实施路线

### 第一阶段：定位和演示口径切换

- 将 README、页面标题、登录页、导航和演示数据切换为企业经营语境。
- 保留现有收支、账户、预算、报表能力，先把分类和样例数据替换为企业场景。
- 将“用户管理”改名为“人员管理”，先承载员工账号、角色、权限和状态。

### 第二阶段：人员模块落地

- 新增员工档案、部门、岗位、入职日期、离职日期、员工状态。
- 增加入职和离职流程页面。
- 将系统登录账号和员工档案建立关联。
- 在报表里加入员工数、人力成本和人员变动。

### 第三阶段：企业财务模型落地

- 将交易重构为企业经营流水，支持客户、供应商、部门、项目和税费字段。
- 引入收入、成本、税费、资产、负债等分类体系。
- 增加应收、应付、开票、收票和付款状态。
- 预算从个人分类预算升级为公司、部门和项目预算。

### 第四阶段：审批、税务和经营控制

- 增加报销、付款、预算调整、入职、离职审批流。
- 增加税期、税费估算和申报提醒。
- 增加审计日志和敏感操作追踪。
- 增加现金流预测、runway 和经营风险预警。

### 第五阶段：多公司与地区政策

- 增加公司选择器和多公司列表。
- 按公司隔离账户、流水、预算、人员、税费和报表。
- 引入地区、政策画像、政策版本和公司级覆盖项。
- 增加跨公司合并报表、公司对比报表和政策变更影响评估。

## 10. 当前实现的过渡策略

当前代码仍以个人记账模型为基础。短期内可以通过下面方式快速过渡：

- `Ledger` 暂时解释为公司经营账本。
- `User` 暂时承担员工账号。
- `Category` 暂时承担收入类型、成本类型和税费类型。
- `TransactionRecord` 暂时承担经营流水。
- `Budget` 暂时承担公司或成本类型预算。
- `RecurringItem` 暂时承担周期性经营事项。

这能快速形成企业版演示，但长期需要按第 6 节重构领域模型，避免个人记账概念继续绑住企业业务。

多公司与地区政策完整规划见 [ENTERPRISE_MULTI_COMPANY_POLICY.md](ENTERPRISE_MULTI_COMPANY_POLICY.md)。
