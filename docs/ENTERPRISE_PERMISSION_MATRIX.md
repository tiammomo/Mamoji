# Mamoji 企业权限矩阵

> 本文定义企业版的人员权限模型。后端接口 `/api/v1/enterprise/permission-matrix` 会返回同一套角色、范围和权限点，前端人员管理页面直接消费该矩阵。

## 1. 设计原则

- 角色决定能做什么，范围决定能看哪些数据。
- 多公司场景下，权限还需要绑定可访问的公司集合。
- 员工档案和系统账号解耦：员工是企业人员对象，账号只是登录凭证。
- 离职员工默认转为只读留档，不再拥有业务写权限。
- 财务、人事、部门负责人权限分离，避免初创公司里“所有管理员都能看所有敏感数据”。
- 创始人保留最终权限分配能力。

## 2. 企业角色

| 角色 Key | 名称 | 默认范围 | 说明 |
| --- | --- | --- | --- |
| `founder` | 创始人/CEO | `company` | 公司所有者，拥有全部经营和权限控制能力。 |
| `finance_admin` | 财务管理员 | `company` | 管理收入、成本、账户、税费、预算和经营报表。 |
| `hr_admin` | 人事管理员 | `company` | 管理员工档案、入职、离职、人力成本和人员权限。 |
| `department_manager` | 部门负责人 | `department` | 管理本部门人员、预算、成本和审批。 |
| `employee` | 普通员工 | `self` | 查看和维护本人资料，提交报销和查看本人审批。 |
| `viewer` | 只读观察者 | `readonly` | 用于投资人、顾问、审计或离职留档的只读访问。 |

## 3. 数据范围

| 范围 Key | 名称 | 说明 |
| --- | --- | --- |
| `group` | 多公司集团 | 可访问同一经营主体下的多家公司汇总数据。 |
| `company` | 全公司 | 可访问公司级数据。 |
| `company_set` | 指定公司集 | 仅访问被授权的多个公司主体。 |
| `department` | 本部门 | 仅访问员工所属部门或负责部门数据。 |
| `self` | 本人 | 仅访问本人档案、单据和审批。 |
| `readonly` | 只读 | 可查看但不可编辑。 |

## 4. 权限点

| 权限 Key | 名称 |
| --- | --- |
| `company.switch` | 切换公司主体 |
| `company.create` | 新增公司主体 |
| `company.manage` | 公司资料管理 |
| `policy.read` | 查看地区政策画像 |
| `policy.manage` | 维护政策配置 |
| `people.read` | 查看人员档案 |
| `people.write` | 维护人员档案 |
| `people.offboard` | 办理离职 |
| `finance.read` | 查看财务数据 |
| `finance.write` | 维护经营流水 |
| `budget.manage` | 预算管理 |
| `tax.manage` | 税费管理 |
| `approval.manage` | 审批处理 |
| `reports.read` | 查看经营报表 |
| `admin.permissions` | 权限分配 |

## 6. 当前落地状态

- 员工档案已新增 `accessRole` 和 `accessScope`。
- 公司主体已新增地区与政策画像字段。
- 企业接口支持公司列表、新增公司，以及部分接口通过 `companyId` 指定公司主体。
- SQLite 会自动为旧员工补齐权限字段。
- 人员管理页支持查看和编辑企业角色、数据范围。
- 权限矩阵已通过 `/api/v1/enterprise/permission-matrix` 返回。
- 当前写操作仍处在过渡期，人员写权限已识别 `founder` 与 `hr_admin`；后续经营流水、税费和审批模块会逐步按该矩阵接入统一鉴权。
