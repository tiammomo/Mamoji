# Mamoji 业务理解

> 本文基于清理前的当前项目代码、接口、前端页面与既有文档抽取，记录 Mamoji 的旧版个人/家庭记账实现基线。项目后续定位已经调整为企业经营记账与人员经营管理系统，新的目标方向以 [ENTERPRISE_PRODUCT_POSITIONING.md](ENTERPRISE_PRODUCT_POSITIONING.md) 为准。

## 0. 定位调整说明

Mamoji 后续不再以个人或家庭记账作为产品目标。新的目标用户是初创公司团队，核心场景包括员工入职、离职、人员档案、权限管理、企业成本支出、收入、税费、预算、现金流和公司经营状态管理。

本文以下内容保留为当前代码和过渡模型说明，可用于判断哪些能力能复用、哪些领域模型需要重构。

## 1. 一句话定位

旧版 Mamoji 是一个面向个人与家庭的记账、预算管理和收支分析系统。企业版改造后，它应演进为初创公司轻量经营管理系统，统一覆盖人员、收入、成本、税费、预算、现金流和经营风险。

## 2. 目标用户与核心价值

### 目标用户

- 个人用户：记录收入/支出、管理账户余额、查看月度/年度财务概览、跟踪预算执行。
- 家庭用户：通过共享账本与成员角色管理家庭收支，理解家庭整体消费结构。
- 管理员用户：维护系统用户、角色和权限，辅助演示、运营或家庭内部账号管理。

### 核心价值

- 快速记账：支持收入、支出、退款流水，按日期、分类、类型筛选。
- 预算反馈：预算与支出联动，实时计算已用、剩余、使用率和风险等级。
- 财务洞察：提供月度概览、趋势、分类占比、年度报表、资产负债、同比环比和异常洞察。
- 家庭协作：通过账本和成员角色承载共享记账场景。

## 3. 角色、权限与访问边界

### 用户角色

- 管理员：角色值为 `1`，可进入用户管理，具备用户、账户、分类、预算管理能力。
- 普通用户：角色值为 `2`，默认无管理权限，可使用自身数据相关的核心功能。

### 权限位

系统使用整数位掩码表达权限：

- `1`：用户管理
- `2`：账户管理
- `4`：分类管理
- `8`：预算管理

账户、分类、预算写操作会校验管理员角色或对应权限位。普通注册用户默认没有这些管理权限，管理员默认拥有全部权限。

### 账本角色

账本成员角色包括：

- `owner`：账本所有者，拥有最高权限。
- `admin`：可添加成员。
- `editor`：预留编辑角色。
- `viewer`：默认只读角色。

当前实现中，账本创建人会自动成为 `owner`；添加成员需要所有者或账本管理员；移除成员只能由所有者执行，且不能移除所有者。

## 4. 核心业务对象

### 用户

用户是认证和数据归属的基础对象，包含邮箱、密码哈希、昵称、头像、家庭 ID、角色、权限、创建/更新时间。系统以 JWT 做无状态鉴权，注册和登录开放，其余 API 默认需要登录。

头像在前端被编码为 `emoji|color` 的形式，用于设置页和侧边栏展示。

### 账户

账户表示资金容器和余额快照，类型包括：

- 现金 `cash`
- 银行卡 `bank`
- 信用卡 `credit`
- 数字钱包 `digital`
- 投资 `investment`
- 负债 `debt`

账户包含名称、类型、子类型、银行、余额、是否计入净资产、用户 ID、账本 ID、状态。账户汇总输出总资产、总负债和净资产，用于账户页、高级报表资产负债视图。

当前账户余额更接近手工维护的快照，交易写入并未完整同步账户余额。

### 分类

分类为收入/支出提供统一语义，字段包括名称、类型、图标、颜色、家庭 ID、是否系统分类。

默认收入分类：

- 工资、奖金、投资、其他收入

默认支出分类：

- 餐饮、交通、购物、娱乐、居住、生活、医疗、教育、人情、其他支出

系统分类不可删除。自定义分类需要分类管理权限或管理员角色。

### 账本

账本表示一组记账数据的协作范围，包含名称、描述、币种、所有者、是否默认、状态。成员关系由 `ledger_member` 维护。

业务上它用于家庭/多人共享账本，但当前数据查询多数仍以 `userId` 为主，`ledgerId` 在账户、预算等实体中已有字段，但在交易、报表等核心路径中尚未完全统一落地。

### 交易

交易是系统核心流水，类型约定：

- `1`：收入
- `2`：支出
- `3`：退款

关键字段包括用户 ID、家庭 ID、类型、金额、分类、账户、日期、备注、原始交易 ID、已退款金额、是否可退款、命中预算 ID。

交易写入规则：

- 金额必须大于 0，最多两位小数，不能超过 10,000,000。
- 日期不能是未来日期，不能早于当前日期 20 年以上。
- 备注最长 200 字符。
- 分类必须存在，并且分类类型必须与交易类型一致。
- 新增/更新支出时会尝试匹配当前有效预算，优先匹配分类预算，再匹配总预算。
- 退款交易不可编辑、不可删除；已经有退款历史的原始支出也不可删除。

当前实现缺口：

- 前端交易表单提交 `accountId`，但后端创建/更新交易时未完整写入账户字段，响应中账户信息也使用了占位 `Cash`。
- 交易写入目前未校验账户归属，也未同步账户余额。

### 退款

退款是支出的反向调整，但以独立交易类型 `3` 保存，保留审计链路：

- 只能对支出发起退款。
- 退款金额必须大于 0，且不能超过剩余可退款金额。
- 退款日期不能早于原始交易日期。
- 原始交易累计 `refundedAmount`。
- 退款交易记录 `originalTransactionId`。
- 预算消耗计算采用有效支出，即支出减去已退款金额，避免预算使用被高估。

### 预算

预算用于期间内支出控制，支持总预算和分类预算：

- `categoryId` 为空：总预算。
- `categoryId` 非空：指定分类预算。

预算字段包括名称、金额、开始日期、结束日期、预警阈值、状态、已用金额、用户 ID、账本 ID、分类 ID。

状态约定：

- `0`：停用/取消
- `1`：进行中
- `2`：已完成
- `3`：超支

预算规则：

- 名称不能为空，最长 64 字符。
- 金额不能为负。
- 起始日期不能晚于结束日期。
- 预警阈值必须在 0 到 100 之间。
- 同一用户、同一分类维度、重叠周期内不能有多个有效预算。
- 支出新增、更新、退款、删除后会刷新受影响预算的已用金额和状态。

预算派生字段：

- `spent`：已用金额。
- `remainingAmount`：剩余金额。
- `usageRate`：使用率。
- `warningReached`：是否达到预警阈值。
- `riskLevel`：`low`、`medium`、`high`、`critical`。
- `riskMessage`：面向前端的可读说明。

## 5. 关键业务流程

### 注册与登录

1. 用户提交邮箱、密码、昵称。
2. 系统检查邮箱唯一性。
3. 密码使用 BCrypt 哈希保存。
4. 登录成功后签发 JWT。
5. 前端保存 token 和用户信息，后续请求带 `Authorization: Bearer <token>`。
6. 后端过滤器解析 token，恢复当前用户上下文。

开发环境曾内置测试账号：

- 邮箱：`test@mamoji.com`
- 密码：`123456`

### 首页概览

首页面向快速感知财务状态：

- 本月收入
- 本月支出
- 本月结余
- 最近交易
- 快捷入口：记账、交易记录、报表、预算

### 记账

1. 前端选择收入/支出、金额、分类、账户、日期、备注。
2. 后端校验类型、金额、日期、分类。
3. 如果是支出，按分类预算优先、总预算兜底的顺序匹配有效预算。
4. 保存交易。
5. 刷新相关预算快照。
6. 返回交易数据和结构化风险评估。

### 退款

1. 前端从可退款支出中选择一笔交易。
2. 用户填写退款金额和退款日期。
3. 后端校验原交易类型、剩余可退金额、退款日期。
4. 更新原交易累计退款金额。
5. 创建退款交易。
6. 刷新预算快照。
7. 返回退款结果和原交易风险状态。

### 预算管理

1. 用户创建预算，设置周期、金额、预警阈值和可选分类。
2. 系统防止同分类同周期预算重叠。
3. 交易写入后预算自动重算已用金额。
4. 前端以进度条、状态标签和预警图标展示预算风险。

### 报表分析

报表从交易、分类、账户、预算聚合数据，帮助用户理解：

- 某段时间收入、支出和结余。
- 收支趋势。
- 收入/支出分类占比。
- 年度每月收支变化。
- 净资产、总资产、总负债。
- 同比和环比变化。
- 最近收支、大额收支、异常分类、预算告警。

## 6. 风控口径

交易风险分为 `low`、`medium`、`high`、`critical`。主要规则：

- 单笔大额支出：支出达到 3,000 记为 `large_expense`，风险升到 `high`。
- 单笔严重支出：支出达到 10,000 记为 `critical_expense`，风险升到 `critical`。
- 无收入支出：当月收入为 0 但已有支出，记为 `expense_without_income`。
- 收支失衡：当月有效支出 / 当月收入达到 1.20，记为 `expense_income_ratio_high`。
- 高频支出：单日支出笔数达到 12，记为 `high_frequency_expense`。
- 疑似重复：同日同分类同金额存在其他支出，记为 `possible_duplicate_expense`。
- 分类突增：同分类本月支出较上月达到 2 倍且增量达到 1,000，记为 `category_expense_spike`。
- 新分类大额：没有上月历史时，当前分类支出达到 3,000 可视为突增。
- 预算观察：预算使用率接近预警阈值前 10 个百分点，记为 `budget_watch`。
- 预算预警：预算使用率达到预警阈值，记为 `budget_warning`。
- 预算超支：预算使用率达到或超过 100%，记为 `budget_overrun`。

风险输出包含：

- 风险等级
- 命中标记
- 风险消息
- 月收入/月有效支出
- 支出收入比
- 单日支出数
- 重复交易数
- 分类本月/上月金额
- 预算额度、已用、剩余、使用率、预警阈值和状态

## 7. 前端功能地图

主导航暴露的主要页面：

- 首页：本月概览、最近交易、快捷入口。
- 新增记账：独立记账入口。
- 交易记录：支出/退款与收入双栏展示，搜索、日期筛选、分页、记账、编辑、退款。
- 统计报表：概览、年度报告、资产负债、异常洞察集中在同一报表工作台中。
- 账户管理：账户列表、账户搜索、账户新增、删除、总资产/总负债/净资产。
- 预算管理：预算列表、日期筛选、搜索、新增、编辑、删除、进度与预警。
- 设置：个人信息、密码、主题、语言，以及低频的数据与管理入口。

设置内保留的低频入口：

- 数据备份：数据量统计、导出、导入文件校验。
- 用户管理：管理员维护用户、角色和权限。

已从主导航隐藏的预留能力：

- 收据管理：图片上传、列表、预览、下载、删除。
- 定期记账：周期项列表、新建、编辑、启停、删除、手动执行。

## 8. 后端接口地图

主要接口模块：

- `/api/v1/auth`：注册、登录、当前用户、资料更新、密码修改。
- `/api/v1/accounts`：账户 CRUD、账户汇总。
- `/api/v1/categories`：分类列表、分类 CRUD。
- `/api/v1/budgets`：预算 CRUD、生效预算、日期范围查询。
- `/api/v1/transactions`：交易列表、新增、更新、删除、可退款列表、退款。
- `/api/v1/ledgers`：账本 CRUD、成员列表、添加成员、移除成员、默认账本。
- `/api/v1/stats`：概览、趋势、分类、年度、资产负债、同比环比、高级洞察。
- `/api/v1/backup`：数据量状态、JSON 导出、导入文件格式校验。
- `/api/v1/recurring`：定期记账内存 CRUD、启停、手动执行。
- `/api/v1/admin/users`：管理员用户管理。

## 9. 数据与归属关系

核心表和实体：

- `users`
- `account`
- `categories`
- `transactions`
- `budget`
- `ledger`
- `ledger_member`

数据归属现状：

- 交易、账户、预算主要按 `userId` 隔离。
- 分类支持系统分类和家庭维度分类。
- 用户保留 `familyId`，用于兼容旧家庭模型。
- 账本模型已存在，但不是所有查询路径都围绕 `ledgerId` 聚合。
- 报表当前主要从当前用户的交易和账户聚合，而非完整家庭/账本聚合。

## 10. 备份与数据导入

备份状态统计：

- 用户数
- 账户数
- 分类数
- 交易数
- 预算数
- 账本数

导出能力：

- 导出当前用户账户、分类、近十年交易、预算和账本为 JSON。

## 11. 定期记账与收据能力现状

### 定期记账

支持周期：

- 每天
- 每周
- 每月
- 每年

记录字段：

- 名称、类型、金额、周期类型、间隔、周几、每月几号、每年几月、开始/结束日期、上次/下次执行日期、状态、执行次数、备注。

### 收据管理

- 图片拖拽/选择上传。

- 404 时提示后端收据模块未启用。

## 12. 主要实现缺口与重建提醒

若后续重建系统，应优先处理这些业务一致性问题：

- 交易和账户余额未形成完整联动；需要明确交易是否自动影响账户余额。
- 交易写入没有完整保存/校验 `accountId`，前后端口径不一致。
- 账本模型存在，但报表、交易、预算等核心聚合仍主要按用户维度，需要统一家庭账本边界。
- 账本列表当前只列出自己拥有的账本，成员可访问详情但列表不一定包含成员账本。
- 分类管理只校验家庭归属和系统分类，缺少更完整的账本/家庭边界策略。
- 周期记账是内存实现，手动执行未真正生成交易。
- 收据模块只有前端和 API client，后端未实现。
- 备份导入只做文件校验，未真正导入数据。
- 默认开发数据和 H2/MySQL 配置口径存在差异，需要在重建时明确开发、测试、生产环境的数据源策略。

## 13. 可延续的产品方向

- 让记账更低摩擦：模板、最近分类、常用金额、定期记账真实落表。
- 让预算更可行动：预算调整建议、异常支出复盘、超支预测。
- 让家庭账本更完整：成员共享视图、权限矩阵、账本级报表。
- 让报表指标沉淀：统一收入、支出、退款、有效支出、结余、净资产口径。
- 让数据更可迁移：可验证导出、幂等导入、备份恢复日志。
