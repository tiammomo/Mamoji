# Mamoji 多公司与地区政策规划

> 本文定义企业版从单公司主体升级到多公司、多地区、政策差异化管理的产品与技术方向。政策内容只作为配置与计算框架承载，不在代码中硬编码具体税率、社保、公积金或地方补贴规则。

## 1. 业务背景

初创团队可能同时运营多个公司主体，例如主体公司、分公司、异地销售主体、境外主体或未来融资/业务拆分后的多个法人。不同主体可能处于不同地区，税务、人事、社保、公积金、发票、申报日历和经营合规要求都不完全一致。

因此，Mamoji 不能长期把“公司”当成一个全局默认值。更准确的抽象是“主体”：公司主体、家庭主体、未来的个体工商户或境外主体都应进入同一套主体切换与授权框架。公司主体、家庭主体、地区和政策版本必须成为核心维度。

## 2. 核心原则

- 一个账号可以访问多家公司，但每次操作必须明确落到某个公司主体。
- 公司注册地、实际经营地、员工工作地和社保缴纳地需要分开建模。
- 政策必须有地区、适用对象、生效时间、失效时间、来源和版本。
- 经营数据按公司隔离，报表可在授权范围内跨公司汇总。
- 家庭资金与公司资金必须分账管理；两者之间的资金流动必须记录为主体间往来。
- 政策计算只能使用已确认的政策版本；未确认政策只能作为提醒或待补全事项。
- 地方政策变化频繁，系统应支持人工确认、覆盖和审计留痕。

## 3. 产品能力

### 多公司主体

- 公司列表：展示用户可访问的所有公司主体。
- 公司切换：工作台、人员、预算、流水、税费、报表都按当前公司过滤。
- 新增公司：维护公司名称、统一社会信用代码、行业、币种、纳税人类型、注册地和经营地。
- 公司状态：筹备中、正常运营、暂停运营、注销/历史留档。
- 集团视图：创始人或授权财务可查看多公司合并收入、成本、现金、税费和人员成本。

### 家庭主体与主体间往来

- 家庭主体：用于管理家庭现金、银行卡、投资、负债、房产、车辆和家庭预算。
- 主体切换：顶部主体选择器应同时列出公司主体和家庭主体。
- 家庭资产页：家庭主体下优先进入资产、账户、家庭流水、家庭预算和净资产视图。
- 主体间往来：家庭与公司发生资金流动时，不能只记成普通转账，需要记录 `from_entity_id`、`to_entity_id`、金额、日期、类型和备注。
- 往来类型：股东垫资、公司归还借款、家庭借款给公司、公司向家庭分红、代垫报销、家庭消费误刷公司卡等。
- 双边留痕：主体间往来应在家庭主体和公司主体两边都能查到，并能关联到实际账户流水。
- 合规提示：公司资金进入家庭或家庭资金进入公司，可能涉及股东往来、借款、分红、报销和税务处理，需要保留用途、凭证和审批记录。

### 地区信息

- 注册地区：国家、省/州、市、区县、注册地址、主管税务机关。
- 经营地区：实际经营区域，可不同于注册地。
- 员工工作地：影响薪资、考勤、社保、公积金和福利口径。
- 社保/公积金缴纳地：可以与工作地不同，需要单独记录。
- 发票和税务地区：影响申报日历、税种、附加税、发票和税务提醒。

### 政策画像

- 税务政策画像：纳税人类型、税种、申报周期、截止日、预估规则。
- 人事政策画像：最低工资、社保公积金基数口径、个税代扣代缴、假勤规则。
- 财务合规画像：会计期间、币种、发票规则、经营许可证/年报提醒。
- 政策版本：按地区和生效期维护，不覆盖历史计算。
- 企业覆盖项：允许财务或 HR 对公司实际执行口径做覆盖，但必须记录原因和操作人。

## 4. 数据模型建议

第一阶段在现有 `companies` 上扩展地区和政策画像字段：

| 字段 | 说明 |
| --- | --- |
| `entity_type` | 主体类型，`company` 表示公司主体，`household` 表示家庭主体。 |
| `country` | 国家或地区，例如中国。 |
| `province` | 省/州。 |
| `city` | 城市。 |
| `district` | 区县。 |
| `registered_address` | 注册地址。 |
| `operating_region` | 实际经营区域展示名。 |
| `tax_authority` | 主管税务机关或办税区域。 |
| `policy_profile_key` | 当前政策画像 Key。 |
| `fiscal_year_start_month` | 财年起始月份，默认 1。 |

第二阶段新增多公司和政策表：

- `company_groups`：同一经营团队或集团。
- `companies`：公司主体，归属 `company_group`。
- `company_memberships`：用户在不同公司的角色、范围和状态。
- `regions`：国家、省、市、区县层级地区字典。
- `policy_profiles`：地区政策画像，关联地区、行业、纳税人类型和生效期。
- `policy_versions`：政策版本快照，记录来源、发布时间、生效期、状态。
- `policy_rules`：可计算规则，按税务、人事、社保、公积金、申报日历等分类。
- `company_policy_overrides`：公司自定义覆盖项。
- `employee_locations`：员工工作地、社保地、公积金地和生效期。
- `entity_transfers`：主体间往来，记录家庭与公司、公司与公司之间的资金流动。
- `policy_audit_logs`：政策同步、确认、覆盖、回滚的审计记录。

第三阶段把现有经营数据全部补齐 `company_id`：

- `company_accounts`
- `business_transactions`
- `budgets`
- `tax_items`
- `payroll_items`
- `approval_requests`
- `audit_logs`

## 5. 权限设计

多公司后，权限需要从 `role + scope` 扩展为 `role + scope + company_set`。

| 范围 | 说明 |
| --- | --- |
| `group` | 多公司集团级，可看合并数据。 |
| `company` | 单公司全量数据。 |
| `company_set` | 指定若干公司。 |
| `department` | 指定公司内的部门。 |
| `self` | 本人数据。 |
| `readonly` | 只读观察。 |

新增权限点：

- `company.switch`：切换当前公司主体。
- `company.create`：新增公司主体。
- `policy.read`：查看政策画像和政策版本。
- `policy.manage`：维护政策配置和企业覆盖项。

## 6. 计算与审计

政策计算不应直接写死在业务代码里，而应走“政策版本 + 规则参数 + 企业覆盖项”的计算链路。

建议计算结果保存这些字段：

- `company_id`
- `policy_profile_key`
- `policy_version_id`
- `calculation_basis`
- `calculated_amount`
- `manual_adjustment`
- `operator_user_id`
- `calculated_at`

这样未来政策调整时，历史税费、工资和预算不会被新规则重新污染。

## 7. 页面规划

- 公司选择器：放在顶部导航或工作台标题区。
- 公司管理页：公司主体、地区、税务画像、经营状态。
- 政策画像页：按公司展示当前适用政策、版本、生效期和待确认项。
- 集团经营看板：跨公司合并收入、成本、税费、现金和人员成本。
- 员工地点页：维护员工工作地、社保地、公积金地和变更历史。
- 税费日历：按公司和地区展示申报截止日与待办。

## 8. 落地路线

### 第一阶段：多公司基础

- `companies` 增加地区与政策画像字段。
- 新增公司列表和新增公司 API。
- 企业接口支持可选 `companyId`。
- 前端增加公司选择器，当前页面按选择的公司过滤。

### 第二阶段：地区政策画像

- 增加 `regions`、`policy_profiles`、`policy_versions`、`policy_rules`。
- 税费事项引用政策版本。
- 人员成本拆分工作地、社保地、公积金地。
- 增加政策待确认和人工覆盖流程。

### 第三阶段：多公司经营闭环

- 经营流水、账户、预算、报表全部公司化。
- 增加集团合并报表和公司对比报表。
- 增加跨公司权限、审计和导出。

### 第四阶段：政策自动化

- 接入官方政策来源或人工维护后台。
- 建立政策变更提醒、版本差异对比和批量影响评估。
- 支持不同地区税费、薪资和社保公积金的规则化计算。

## 9. 风险边界

- 系统可以做政策配置、提醒和估算，但最终申报口径需要由财务或专业顾问确认。
- 对地方政策、税率、社保公积金基数等高频变化内容，必须记录来源和确认时间。
- 没有政策版本的数据不能自动生成正式税费或薪资结果，只能展示为待确认。
